# evaluate

基于双源事实的严格评估。所有结论必须有来源、有推理。

## 核心原则

评估不能只靠外部搜索，也不能只看项目代码。需要从两个维度同时收集事实，交叉验证：

- **项目事实**：实际代码、依赖版本、配置、git 历史、项目约定 — 代表"这里实际是什么"
- **外部事实**：最佳实践、官方规范、社区共识、benchmark — 代表"应该是什么"

每个判断都要能追溯到具体来源，每个推理链条都要显式展示。

## 执行步骤

### 1. 解析评估目标

从 `$ARGUMENTS` 和对话上下文中提取：

- **评估对象**：具体是什么（库、工具、方案、声明、代码模式等）
- **评估维度**：用户关心什么（性能？安全？维护性？准确性？性价比？）
- **上下文信息**：对话中已有的相关信息

如果评估目标不明确，向用户确认后再继续。

### 2. 收集项目事实

根据评估对象，收集项目内可验证的事实。不是所有项都适用，按需选择：

**代码事实**
- 读取评估对象涉及的源码，理解实际实现
- 检查调用链、依赖关系、使用方式
- 统计相关代码量（文件数、函数数、行数）

**依赖事实**
- 检查 `package.json`、`go.mod`、`Cargo.toml`、`pyproject.toml` 等，确认依赖版本
- 检查锁文件（`package-lock.json`、`go.sum` 等）确认实际安装版本
- 检查是否有已废弃或已知有漏洞的依赖

**配置事实**
- 项目的 linter、formatter、CI 配置
- `CLAUDE.md`、`.claude/rules/` 中的项目约定
- `.env.example`、`config/` 目录中的配置模板

**历史事实**
- `git log --oneline` 查看评估对象相关的变更历史
- `git blame` 了解代码的年龄和修改频率
- 相关的 PR、issue、commit message 中的决策背景

### 3. 收集外部事实

这一步不可跳过。搜索策略：

**第一轮：广度搜索**

根据评估对象，选择合适的搜索工具组合：

- `mcp__plugin_ecc_exa__web_search_exa` — 适合技术博客、对比文章、使用经验
- `WebSearch` — 适合官方文档、最新动态、已知关键词
- `mcp__plugin_ecc_context7__resolve-library-id` + `mcp__plugin_ecc_context7__query-docs` — 如果涉及具体库/框架，查官方文档

搜索查询构造技巧：
- 中英文各搜一轮（技术内容英文覆盖面更广）
- 包含版本号或年份（如 "2025 2026"）确保时效性
- 加入 "vs" "comparison" "benchmark" "review" 等对比性关键词
- 搜索对象的上游依赖和下游用户评价

**第二轮：深度搜索**

根据第一轮结果，针对关键争议点或信息缺口补充搜索：

- 搜索反面证据（"X problems" "X issues" "X limitation"）
- 搜索 GitHub issues、Stack Overflow 上的实际使用反馈
- 搜索竞品对比和迁移经验

**第三轮：交叉验证**

将项目事实和外部事实对比，发现矛盾或缺口时补充搜索：

- 项目实际做法 vs 业界推荐做法的差异
- 项目使用的版本是否存在已知问题
- 项目的用法是否符合库/框架的官方推荐

### 4. 整理证据

将双源事实按评估维度整理，标注来源类型：

```
## 证据整理

### 维度 A：{如 安全性}

[项目事实]
- 项目使用 X v2.3.1，配置了 Y（来源：package.json:12）
- 代码中 Z 处存在未验证输入（来源：src/handler.ts:45）

[外部事实]
- X v2.3.1 存在已知 CVE-2025-XXXX（来源：url）
- 官方文档推荐使用 W 替代 Y（来源：url）

[矛盾/缺口]
- 项目做法 A 与官方推荐 B 不一致，原因是 ...
```

### 5. 严格评估

基于双源证据进行评估，遵循以下规则：

**推理透明**：每个结论展示完整推理链条：
```
结论：X 在 Y 场景下优于 Z
推理：
  1. [项目] 当前代码使用 Z，覆盖了 N 个文件（来源：git grep）
  2. [外部] X 的 benchmark 数据为 ...（来源：url）
  3. [外部] Z 在同样场景下的数据为 ...（来源：url）
  4. [外部] 但 X 在 ... 条件下会退化（来源：url）
  5. [推断] 在项目的 Y 场景中，X 更优，因为 ...
```

**区分来源类型**：
- `[项目]` — 来自项目代码、配置、历史的陈述
- `[外部]` — 有明确 URL 或文档引用的外部事实
- `[推断]` — 基于事实的推理，说明推断依据
- `[不确定]` — 信息不足无法判断，明确说明

**反面证据必须呈现**：如果搜索到了不利于最终结论的证据，必须列出并解释为什么没有采纳。

### 6. 输出评估报告

使用以下格式：

```markdown
# 评估：{评估对象}

## 结论摘要

{一句话结论，如果需要前提条件要说明}

## 评估维度

### {维度 1}
- **评级**：{优秀 / 良好 / 一般 / 较差 / 危险}
- **判断**：{具体说明}
- **证据**：
  - [项目] {项目内事实，附文件路径或行号}
  - [外部] {外部事实，附 url}
  - [推断] {推理过程}

### {维度 2}
...

## 争议与风险

{双源证据之间的矛盾、未确认信息、潜在风险}

## 局限性

{本次评估的局限：搜索覆盖面、项目事实的完整度、时效性}
```

### 7. 给出建议（如果适用）

如果评估发现了问题或改进空间，给出可操作的建议：

- 具体到文件和行号的修改建议
- 优先级排序（哪些应该立即改，哪些可以后续处理）
- 每个建议附上理由和来源

## 事实收集量指南

根据评估的复杂度调整收集力度：

| 复杂度 | 项目事实 | 外部搜索 | 示例 |
|--------|----------|----------|------|
| 简单验证 | 读相关代码 | 2-3 次搜索 | "这个 API 还支持吗"、"这个参数的默认值" |
| 方案评估 | 读代码+配置+依赖 | 4-6 次搜索 | "A 库 vs B 库"、"这个架构模式好不好" |
| 深度评估 | 读代码+配置+依赖+历史 | 6-10 次搜索 | "技术选型评估"、"安全性全面审查" |

宁可多收集，不要漏掉。事实收集成本低，错误结论代价高。

## 常见评估场景

### 技术选型评估

- [项目] 当前用什么、版本、依赖深度、替换成本
- [外部] GitHub 活跃度、下载量、已知 issues、社区评价、竞品对比

### 代码实践评估

- [项目] 实际代码怎么写的、影响范围、是否有测试覆盖
- [外部] 官方文档推荐、社区共识、反面案例、版本演进变化

### 声明验证

- [项目] 声明在项目中的体现和上下文
- [外部] 原始来源、可信引用、反驳证据、前提条件

### 架构方案评估

- [项目] 当前架构、模块边界、依赖方向、变更频率
- [外部] 业界同类问题的解决方案、架构模式的优缺点、大规模案例
